## 1. 文档目标

本 PRD 用于指导一个新的初赛 Demo 版本。这个版本不再把“QQ 养虾”表现成独立产品工作台，而是表现成：

> **用户正在用 QQ，小龙虾自然出现在聊天流程中，并以聊天式交互和卡片式结果完成核心能力。**

本文档的目标是把以下方向落实为可演示产品：

1. 主界面先是 QQ，再是小龙虾。
2. 主交互先是聊天，再是功能详情。
3. 工作记录、群聊总结、回复草稿优先以文本卡片形式出现，@ 消息作为群聊总结卡内的可选提醒信号。
4. 日记、成就、空间动态优先以强视觉卡片形式出现。
5. 权限、进群、资料卡、空间作为深层承接，不做主导航中心。

## 2. 初赛版本一句话

QQ 养虾是一只存在于 QQ 里的 AI 小龙虾。它会在你聊天和看群的过程中，自然感知授权群里的重点与 @ 信号、回答追问、生成回复草稿，并在关键时刻把这一天写成一张漂亮的日记卡片，沉淀为成就、挂饰和小龙虾空间动态。

## 3. 初赛版本核心判断

### 3.1 为什么要重构 Demo

上一版 Demo 已经证明了产品方向可行，但问题在于：

1. 结构上更像“QQ 养虾独立产品中心”。
2. 用户需要频繁切页面。
3. 功能入口显得过重。
4. 小龙虾像主角，QQ 像背景。
5. 评委容易理解成“套在 QQ 主题上的 AI 功能集合”。

新版 Demo 的重构目标是：

1. 让评委一眼看出这是 QQ 主应用中的 Agent。
2. 让用户感觉小龙虾和自己是在“聊天”，不是在“操作后台”。
3. 让卡片成为结果承载体，让少量页面成为承接和沉淀体。

### 3.2 与元宝 AI 群消息总结的差异

本 Demo 需要明确表达：QQClaw 不重复做一个“群消息总结按钮”，也不替代元宝 AI。

核心区别是：

1. 元宝 AI 群消息总结更适合用户已经打开群聊后的手动补课。
2. QQClaw 更适合用户授权后的主动巡群提醒，尤其是平时不常打开、但又不想错过的课程群、社团群、比赛群和兴趣群。
3. QQClaw 的职责是发现值得关注的内容、轻量提醒、给出原因，并把用户带回原群查看上下文。
4. 若接入腾讯内部能力，QQClaw 可以调用元宝 AI 的群消息理解/摘要能力，但产品角色仍是个人宠物 Agent 的跨群主动感知和提醒。

Demo 讲解时应强调：

> 元宝 AI 是群内手动总结引擎，QQClaw 是授权后的个人宠物 Agent，主动帮你巡看那些你平时不打开、但不想错过的群。

### 3.3 初赛版本产品本质

在初赛版本里，QQ 养虾不是“页面集合”，而是一个三层体验：

| 层级 | 作用 | Demo 形态 |
| --- | --- | --- |
| QQ 主界面层 | 证明产品原生嵌在 QQ 里 | 消息列表、群聊窗口、输入框、群资料 |
| Agent 交互层 | 证明 AI 真的在工作 | 小龙虾私聊窗、群聊顶部提示、文本卡片、浮层抽屉 |
| 关系沉淀层 | 证明陪伴和社交价值 | 日记卡片、成就 toast、小龙虾资料卡、小龙虾空间 |

### 3.4 初赛版本必须传达的产品感

评委体验后，必须得到这几个印象：

1. 这不是一个通用 AI 助手，而是 QQ 里的小龙虾伙伴。
2. 这不是纯宠物，而是真能帮我捞消息、补群聊、给建议。
3. 它出现得很自然，像 QQ 里的一个原生助手，而不是另一个独立 App。
4. 它不仅有用，还会留下日记、成就和分享内容。
5. 它很克制，不会偷看、不乱说、不乱发。
6. 它的群聊能力不是手动总结替代品，而是授权后的主动巡群提醒。

## 4. 初赛目标与边界

### 4.1 目标

初赛版本的目标不是实现完整产品，而是用最小而完整的演示闭环证明：

1. QQ 场景适配性。
2. AI 原生性。
3. 情绪陪伴和长期成长价值。
4. 权限可控与落地可行性。

### 4.2 初赛闭环

用户打开 QQ -> 在消息列表里看到“小钳”会话或重点提醒 -> 打开 `软件工程课设群` -> 看到群聊顶部的小龙虾提示卡 -> 点击进入小龙虾交互层 -> 查看群聊感知卡和 @ 信号 -> 追问“我需要回什么” -> 看到回复草稿 -> 问“小龙虾今天做了什么” -> 得到工作记录卡 -> 在合适时机弹出日记卡片 -> 解锁成就和挂饰 -> 分享卡片给好友形成传播 -> 用户点击小龙虾头像进入资料卡和空间 -> 小龙虾把日记或成就发布为自己的空间动态 -> 好友或好友的龙虾在空间动态下互动评论 -> 小龙虾生成回复草稿 -> 在群设置看到“邀请我的小龙虾进群”的权限预览。

### 4.2.1 兴趣系统首版范围

兴趣系统第一版不同时铺满所有兴趣类型。Demo 主打一个纵深样例：**音乐**。

音乐纵深用于展示小龙虾如何围绕用户已选择、已表达或已授权的兴趣，生成陪伴式提醒、解释来源，并在后续版本承接 QQ 音乐、龙虾空间动态、成就和挂饰。

同时保留“兴趣社群推荐”作为通用能力，但首版只用一张羽毛球公开社群资料 mock 卡做演示样例，用来说明 QQ 的兴趣同好发现能力。它不是第一版主线，也不扩展成复杂多兴趣调度。

Demo 初始兴趣画像可以预置，但语义必须是：这些是用户已经选择、已经表达、已经授权或已经设置过的模拟状态，不是小龙虾凭空知道用户隐私。每条兴趣画像都必须说明“为什么知道”和“来自哪里”。

### 4.3 不做范围

| 不做项 | 原因 | Demo 替代方案 |
| --- | --- | --- |
| 真实 QQ 登录 | 初赛无必要，接入成本高 | 使用模拟 QQ 用户和群聊 |
| 真实群聊数据读取 | 涉及权限、接口和隐私 | 使用结构化 mock 群聊数据 |
| 真实 QQ 音乐 API | 初赛只验证兴趣纵深表达 | 使用模拟 QQ 音乐授权数据 |
| 真实 QQ 群搜索 | 涉及社群隐私和推荐合规 | 使用公开社群资料 mock 卡 |
| 真实发消息 | 高风险 | 只展示草稿和待确认状态 |
| 真实发空间动态 | 非必要 | 展示动态预览和空间预览 |
| 自动加群、申请入群或私聊群主 | 高风险动作 | 只展示推荐理由和用户确认入口 |
| 自动替用户评论或申请加入 | 高风险动作 | 只生成预览草稿，必须用户确认 |
| 复杂多兴趣调度和完整推荐算法 | 超出首版范围 | 只做音乐主线，羽毛球作为社群推荐 mock 样例 |
| 真实多用户互动链路 | 复杂度过高 | 使用预设好友评论和访客 |
| 无限重生成日记 | 会弱化日记的记忆属性 | 只允许查看、隐藏敏感信息、切换卡片版 |
| 完整 QQ 秀系统 | 超出初赛范围 | 轻量形象 + 挂饰状态变化 |

## 5. 初赛产品形态

### 5.1 主界面原则

初赛 Demo 的第一屏必须是一个 **QQ 主界面**，而不是功能导航页。

推荐布局：

```text
左侧：消息列表 / 会话列表
中间：当前群聊内容区
右侧：群资料 / 文件 / 成员信息
顶部：当前群标题、在线状态、轻量功能按钮
底部：聊天输入框
附加层：小龙虾提示条、小龙虾私聊抽屉、小龙虾卡片、日记弹卡
```

### 5.2 入口层级

#### 一级入口：用户正在使用 QQ 时自然看到的入口

1. 群聊顶部提示条。
2. 输入框旁小龙虾按钮。
3. 消息列表中的小红点提示。
4. 与“小钳”的私聊会话。
5. 小龙虾头像。

#### 二级入口：轻量展开层

1. 小龙虾抽屉。
2. 文本卡片堆叠区。
3. 日记弹卡。
4. 成就 toast。
5. 小龙虾资料卡。

#### 三级入口：完整承接页

1. 小龙虾空间页。
2. 权限设置页。
3. 群设置里的小钳模块。

## 6. 初赛信息架构

### 6.1 主导航应取消什么

初赛版本不应再把这些做成左侧一级主导航：

1. 工作记录。
2. 日记。
3. 成就。
4. 分享。
5. 空间。
6. 进群。
7. 权限。

这些内容的存在方式应调整为：

| 内容 | 新入口 |
| --- | --- |
| 工作记录 | 小龙虾对话中的文本卡片 |
| 日记 | 事件触发弹卡 |
| 成就 | toast、小卡片、资料卡 |
| 分享 | 日记卡片上的操作 |
| 空间 | 小龙虾头像或卡片入口 |
| 进群 | 群设置中的小钳模块 |
| 权限 | 小龙虾设置 / 群设置 |

### 6.2 初赛必要视图

尽管不做功能主导航，但仍需实现这些核心视图：

| 视图 | 类型 | 作用 |

| QQ 主界面壳子 | 主场景视图 | 演示小龙虾嵌入 QQ |
| 小龙虾私聊窗 | 对话视图 | 展示聊天式 Agent |
| 小龙虾抽屉 / 浮层 | 轻量视图 | 展示文本卡片和上下文结果 |

| 权限设置页 | 深度承接页 | 展示授权与日志 |
| 群设置小钳模块 | 深度承接页 | 展示龙虾进群权限预览 |

## 7. 用户流程

### 7.1 主要演示流程

1. 用户进入 QQ 主界面，当前打开 `软件工程课设群`。
2. 消息列表中已经能看到“小钳”专属会话或“我帮你捞到重点了”的提示。
3. 群聊顶部出现一条提示：`小钳帮你捞到 3 条重点，其中 1 条需要今天回复。`
4. 用户点击提示条，右侧展开小龙虾抽屉。
5. 抽屉中先展示：
   - 群聊总结卡片（可切换多个群，含相关 @ 和摘要）
   - 回复草稿卡片
6. 用户继续在“小钳”私聊窗输入：`我需要回什么？`
7. 小龙虾返回一张更具体的回复草稿卡片。
8. 用户继续问：`你今天做了什么？`
9. 小龙虾返回工作记录卡片。
10. 当用户处理完重点消息后，界面触发一张日记弹卡。
11. 日记卡片上展示小龙虾日记、配图、金句和分享按钮。
12. 同时解锁“消息打捞员”成就，小龙虾获得“小红旗”挂饰。
13. 用户点击分享卡片，把内容发给好友，好友因此被引导到“小虾空间”。
14. 用户点击小龙虾头像，打开资料卡，看到当前挂饰和最近成就。
15. 用户再点击“我的小虾空间”，进入空间页，查看动态、日记时间线和勋章墙。
16. 小龙虾把日记或成就发布成空间动态，好友和好友的龙虾在空间动态下互动评论。
17. 小龙虾生成评论回复草稿。
18. 最后用户在群设置中看到“邀请我的小龙虾进群”，打开权限说明和群管理确认流程。

### 7.2 次要探索流程

1. 用户从消息列表点击“小钳”私聊。
2. 用户主动问：
   - `刚才群里讲到哪了？`
   - `有人 @ 我吗？`
   - `帮我写一句轻松一点的回复。`
3. 小龙虾返回对应文本卡片。

### 7.3 情绪触发流程

1. 小龙虾第一次成功捞到重点消息。
2. 用户完成处理后。
3. 系统触发日记卡片和成就提醒。
4. 用户可：
   - 分享
   - 存入空间
   - 收起
   - 查看成就

## 8. 核心交互模型

### 8.1 聊天式交互是第一主交互

用户与小龙虾的关系，优先通过“对话”建立。

用户会像和好友聊天一样问它：

| 用户输入 | 返回结果 |
| --- | --- |
| 今天你帮我做了什么？ | 工作记录卡片 |
| 总结一下课设群刚才说了什么。 | 群聊总结卡片 |
| 有没有人 @ 我？ | 群聊总结卡片中的相关 @ |
| 我需要回什么？ | 回复草稿卡片 |
| 这条评论怎么回？ | 评论草稿卡片 |
| 你今天写日记了吗？ | 日记卡片摘要 |

### 8.2 卡片式结果是第二主交互

小龙虾的输出不是长篇大论，而是卡片。

#### 文本型结果卡片

这些可以是轻量文本卡片，不需要强视觉：

1. 群聊总结卡片。
2. 回复草稿卡片。
3. 工作记录卡片。

这类卡片必须具备：

1. 标题。
2. 摘要。
3. 行动原因。
4. 下一步操作。
5. 继续追问入口。

#### 强视觉结果卡片

这些应有更明显的美术表现：

1. 日记卡片。
2. 成就卡片。
3. 分享卡片。
4. 小龙虾空间动态卡。

### 8.3 先结果，后详情

所有结果都应满足：

1. 第一层一眼能看懂。
2. 第二层可以展开细节。
3. 只有必要时才进入深度页面。

## 9. 功能需求

### 9.1 群聊总结卡与相关 @

#### 目标

让用户在不翻大量消息的前提下看到最重要内容。

本能力应被表达为“群聊总结卡”：QQClaw 在用户授权后整理一个或多个群的重点内容；如果其中存在 @ 用户的消息，就在对应群的卡片页内展示，没有则不显示 @ 区域。

#### 触发方式

1. 用户授权后，小龙虾整理选中的模拟群聊。
2. 群聊顶部出现轻量提示。
3. 消息列表中对应群显示“小钳捞到重点”标记。

#### 输出

群聊总结卡中的相关 @ 区，结构如下；没有相关 @ 时不展示该区：

| 字段 | 内容 |
| --- | --- |
| 标题 | 群聊总结 |
| 群名 | 软件工程课设群 |
| 时间 | 10:17 |
| 发送人 | 组长 |
| 摘要 | @你确认是否负责接口文档 |
| 原因 | 与你直接相关，且影响明晚的分工表提交 |
| 操作 | 跳转原群、查看简短摘要、写回复草稿、降低提醒频率 |

#### 验收

1. 至少展示 3 条重点消息。
2. 至少分出待回复、任务提醒、情绪互动三类。

### 9.2 群聊总结摘要

#### 目标

让用户在看到 QQClaw 的总结后，快速判断是否值得回到原群继续看。它不是替代用户阅读完整群聊，而是服务“整理 -> 判断 -> 跳回原群”的链路。

#### 入口

1. 群聊总结卡中的“查看简短摘要”。
2. 群聊顶部提示卡中的“总结本群”。
3. 输入框旁小龙虾按钮。
4. 与小钳私聊中输入问题。

#### 输出

群聊总结卡的摘要区，结构如下：

1. 本段主要在讨论什么。
2. 与用户有关的内容。
3. 待办事项。
4. 重要时间点。
5. 建议回复。

#### 示例

> 课设群刚才主要在讨论模块分工和分工表提交。  
> 和你有关的是接口文档任务，组长在 10:17 @ 你确认是否负责。  
> 明晚 20:00 前需要提交分工表。  
> 建议你回复：“可以，我负责接口文档。今晚我补一版权限说明，明天一起合进分工表。”  

#### 验收

必须支持至少 3 个预设追问：

1. 这张总结卡里有哪些和我有关的消息？
2. 我需要回什么？
3. ddl 是什么时候？

### 9.3 回复草稿

#### 目标

降低用户处理群聊的表达成本。

#### 入口

1. 群聊总结卡中相关 @ 的“写回复草稿”。
2. 群聊总结卡摘要区的“写回复草稿”。
3. 用户主动问“我需要回什么？”

#### 输出

回复草稿卡片：

| 字段 | 内容 |
| --- | --- |
| 建议回复 | 可以，我负责接口文档。今晚我补一版权限说明，明天一起合到分工表里。 |
| 语气 | 轻松 / 正式 / 简洁 |
| 来源 | 10:17 组长 @你，10:43 同学提醒权限说明可能是重点 |
| 操作 | 插入输入框、换个语气、我自己改 |

#### 风险控制

必须显式显示：

> 这条回复只是草稿，发送前需要你确认。

### 9.4 工作记录

#### 目标

让用户知道小龙虾做过什么，建立信任。

#### 呈现方式

工作记录不做一级页面，而作为对话结果卡片或抽屉卡片出现。

#### 典型触发

用户问：

> 你今天帮我做了什么？

#### 输出

工作记录卡片：

> 队长，我今天巡了 3 个你授权的群，捞到 4 条和你有关的消息。  
> 其中 2 条需要回复，1 条是明晚 20:00 的 ddl，1 条是班级通知。  
> 我生成了 1 次课设群总结、1 条回复草稿和 1 篇日记。  
> 回复草稿已准备好，可以复制后跳回原群使用。  

#### 展开详情

点击“查看详情”可以展示时间线：

1. 10:17 捞到组长 @。
2. 10:45 识别分工表 ddl。
3. 11:20 生成课设群总结。
4. 11:22 生成回复草稿。
5. 23:00 生成今日日记。

### 9.5 日记

#### 目标

让小龙虾的实用行为转化为情绪价值。

#### 基本原则

1. 日记是事件触发，不是页面生成器。
2. 日记是第一人称记忆，不是用户操作报告。
3. 日记是强视觉卡片，而不是普通文本块。

#### 触发时机

初赛版本推荐固定为：

1. 小龙虾第一次成功帮用户捞到重点消息后。
2. 用户已看过总结或回复草稿后。

#### 出现方式

在 QQ 主界面中，以一张漂亮的卡片弹出，类似空间动态卡或内容卡。

卡片包含：

1. 标题。
2. 小龙虾卡通配图。
3. 一句金句。
4. 日记正文摘要。
5. 成就提示。
6. 操作：
   - 分享
   - 存入小虾空间
   - 查看成就
   - 收起

#### 限制

1. 不支持反复重新生成整篇日记。
2. 可以支持：
   - 全文 / 摘要切换
   - 隐藏敏感信息
   - 更换配图
   - 存入空间

#### 日记示例

标题：我第一次从 99+ 里捞到了队长

摘要：

> 今天我第一次正式巡群。课设群像一锅开水，咕嘟咕嘟冒到 99+。我一开始还有点慌，钳子都不知道先夹哪条。后来我发现，有一条 @ 队长的消息被压在中间：明晚 20:00 前要交分工表。  
> 我把它捞出来，夹上小红旗，放进了收件箱。  
> 队长点开的时候只说了一句：“还好你看到了。”  
> 我表面很镇定，其实虾壳都亮了一下。  

### 9.6 成就与挂饰

#### 目标

把小龙虾的工作和用户行为沉淀为成长。

#### 呈现方式

成就不做主页面，分为三层：

1. 即时 toast：`小钳获得「消息打捞员」`
2. 小卡片：点击查看详情
3. 资料卡 / 空间中的长期展示

#### 初赛成就样例

| 触发事件 | 用户成就 | 小龙虾成就 | 挂饰 |
| --- | --- | --- | --- |
| 完成领养 | 第一次领养小龙虾 | 第一次认主 | 初始小背包 |
| 授权群聊 | 第一次带虾巡群 | 第一次出门巡群 | 群工牌 |
| 查看重点消息 | 第一次查看虾虾重点消息 | 第一次捞到 @ | 小红旗 |
| 完成总结 | 第一次补课 99+ | 第一次捞起 99+ | 消息网兜 |
| 生成日记 | 第一次看到小龙虾日记 | 第一次写下今天 | 日记贴纸 |

### 9.7 小龙虾资料卡

#### 目标

作为小龙虾的轻量个人主页。

#### 入口

点击小龙虾头像。

#### 内容

1. 小龙虾形象。
2. 当前挂饰。
3. 今日状态。
4. 最近一条日记摘要。
5. 最近解锁的成就。
6. 去小虾空间。
7. 去权限设置。

#### 作用

它是“成就”和“空间”的统一轻入口，比做一个大成就页更自然。

### 9.8 小龙虾空间

#### 目标

承载长期成长与社交沉淀。

#### 内容结构

1. 空间头图。
2. 小龙虾形象。
3. 由小龙虾发布的今日动态。
4. 日记时间线。
5. 勋章墙。
6. 挂饰展示。
7. 评论区。
8. 小龙虾回复。
9. 好友龙虾的来访或互动痕迹。

#### 空间规则

1. 这是小龙虾自己的 QQ 空间，由小龙虾发动态，不是人类用户自己发动态。
2. 人类用户可以访问、点赞、评论、分享，但不能直接替小龙虾在这个空间发动态。
3. 好友的人类账号可以进入这个空间浏览。
4. 好友拥有的小龙虾也可以在这个空间中互动、评论，形成“龙虾和龙虾”之间的社交感。
5. 分享卡片的作用是把好友引流到这个空间，而不是让评论直接发生在分享卡片本身上。

#### 初赛实现重点

初赛不需要完整空间社交链路，但必须让评委看到：

1. 小龙虾的内容最终会沉淀到一个可访问的空间。
2. 好友可以评论、点赞。
3. 好友的龙虾也可以有互动痕迹或评论示例。
4. 小龙虾可以给出评论草稿。

### 9.9 评论回复

#### 目标

让小龙虾参与社交传播。

#### 入口

小龙虾空间动态收到评论后，用户可点“让小钳回一句”。

#### 输出

评论回复草稿卡片。

#### 示例

好友对空间动态的评论：

> 这虾还挺会写。

小龙虾回复草稿：

> 别夸太大声，我的钳子会翘起来。  

#### 风险控制

必须显示：

> 这条评论只是草稿，发送前需要你确认。

### 9.10 龙虾进群

#### 目标

证明小龙虾未来可以成为 QQ 群中的原生 Agent。

#### 入口

群设置中的“小钳设置”模块。

#### 初赛需要演示的内容

1. 邀请我的小龙虾进群。
2. 小龙虾进群后能做什么。
3. 小龙虾不能做什么。
4. 需要群管理确认。
5. 群成员可见公告。

#### 群公告文案

> 小龙虾已进入本群。它是用户的 AI 社交伙伴，可在授权范围内整理群聊重点、识别 @ 信号和生成回复草稿。回复草稿需要用户复制或确认后再使用。

## 13. 权限与安全

### 13.1 权限原则

1. 默认只使用模拟数据。
2. 按群授权。
3. 高风险动作默认只生成草稿。
4. 工作记录可追问。
5. 分享默认脱敏。
6. 进群默认需要管理员确认。

### 13.2 必须显式出现的文案

1. 我只看你授权的群。
2. 这条回复只是草稿。
3. 这条评论等你确认后才会发出。
4. 这篇日记已隐藏群名和人名。
5. 小龙虾进群需要群主或管理员同意。

## 14. 3 分钟录屏脚本

### 14.1 脚本目标

让评委在 3 分钟内感受到：

1. 入口自然。
2. QQ 场景真实。
3. AI 能力有效。
4. 小龙虾有情绪和成长。
5. 权限边界清楚。

### 14.2 时间分配

| 时间 | 画面 | 讲解重点 |
| --- | --- | --- |
| 0:00-0:20 | QQ 主界面、课设群 | 这不是独立产品，而是 QQ 聊天中的小龙虾 Agent |
| 0:20-0:50 | 小钳群聊总结卡 | 区分元宝 AI 的手动群总结：QQClaw 可以把多个授权群整理进一张可切换卡片 |
| 0:50-1:20 | 群聊简短摘要、跳转原群、回复草稿卡片 | QQClaw 可调用群理解能力，但核心是提醒值得看的内容并把用户带回原群 |
| 1:20-1:40 | 小龙虾私聊窗、工作记录卡片 | 用户可以像和好友聊天一样问它今天做了什么 |
| 1:40-2:10 | 日记弹卡 | 小龙虾把今天的经历写成第一人称日记 |

## 15. 验收标准

### 15.1 产品形态验收

1. 第一屏必须是 QQ 主界面，不是功能首页。
2. 产品主路径必须以聊天式交互推进。
3. 文本型结果卡片必须能独立承担核心信息。
4. 日记必须以弹卡或动态卡方式出现，不是独立生成页。
5. 工作记录不能再是一级页面逻辑。

### 15.2 功能验收

1. 能展示至少一张群聊总结卡片，且卡内可切换多个群；相关 @ 仅在存在时展示。
2. 能展示一张回复草稿卡片。
3. 能展示一张工作记录卡片。
4. 能展示一张日记卡片。
5. 能展示一次成就解锁与挂饰变化。
6. 能展示小龙虾资料卡。
7. 能展示空间页和评论草稿。
8. 能展示进群权限说明。

### 15.3 赛题验收

1. 评委能感知 QQ 场景的适配性。
2. 评委能看到 AI 原生能力，而不是静态脚本。
3. 评委能看到实用价值和情绪价值并存。
4. 评委能看到权限边界与落地思考。

## 16. 初赛版本总结

v2 初赛 Demo 的本质不是“把更多功能做成更多页面”，而是：

1. 把 QQ 主界面做出来。
2. 把小龙虾的入口做自然。
3. 把核心能力收敛成聊天式 Agent 体验。
4. 把结果用卡片承载。
5. 把日记、成就和空间作为陪伴与传播层自然衔接出来。

最终评委看到的应该是：

> 我在用 QQ 的过程中，这只小龙虾像一个恰到好处出现的伙伴。它能帮我处理社交信息流，也能把我们的这一天留下来。
