## 1. 产品重定义

### 1.1 一句话

QQ 养虾不是一个独立的龙虾 App，也不是一个单独的 AI 面板，而是一个 **嵌在 QQ 聊天、群聊、动态、空间和设置流程中的原生 AI 社交伙伴**。

它会在用户使用 QQ 的过程中，以自然、不打扰但随时可唤起的方式出现：

1. 在群聊里帮用户捞出 @、任务和重点消息。
2. 通过聊天式交互回答“刚才发生了什么”“我需要回什么”。
3. 在关键时刻以卡片方式推送小龙虾的工作记录、日记、成就和空间动态。
4. 通过拟人化成长、装扮和空间沉淀，把工具能力转化为长期社交关系。

### 1.2 核心判断

此前版本的问题在于：

1. 信息架构更像“QQ 养虾独立产品后台”。
2. 小龙虾功能被拆成多个一级页面，像功能目录，不像 QQ 里的自然入口。
3. “工作记录”“日记”“成就”“空间”被产品化为大页面，弱化了它们在聊天流程中的即时出现感。

v2 的核心调整是：

1. **主界面先是 QQ，再是小龙虾。**
2. **交互核心先是聊天，再是功能页面。**
3. **结果展示先是卡片，再是详情页。**
4. **信息入口先是场景触发，再是主动探索。**

### 1.3 与元宝 AI 群总结的关系

QQClaw 不应被定义为“再做一个群消息总结按钮”。它和元宝 AI 群总结的关系应是互补：

| 能力 | 典型场景 | 用户动作 | 产品角色 |
| --- | --- | --- | --- |
| 元宝 AI 群总结 | 用户已经打开某个群，想快速补课 | 手动点击总结 | 群内即时阅读辅助 / 总结引擎 |
| QQClaw 小钳巡群 | 用户平时不常打开某些群，但不想错过重点 | 提前授权，等待提醒 | 跨群主动感知的个人宠物 Agent |

因此 QQClaw 的差异化不是替代用户阅读完整群聊，而是：

1. 用户授权后主动巡看指定群。
2. 只在出现 @、截止时间、活动、兴趣话题等值得关注内容时提醒。
3. 用小龙虾语气解释“为什么值得看”。
4. 引导用户跳回原群查看上下文。
5. 可调用元宝 AI 的群消息理解/摘要能力，但负责授权、巡群、提醒判断、工作记录和宠物化表达。

### 1.4 产品本质

QQ 养虾本质上是一个三层结构产品：

| 层级 | 作用 | 呈现形态 |
| --- | --- | --- |
| 底层 | AI Agent 能力层 | 巡群、群聊总结、@ 信号识别、问答、提醒、回复草稿、评论草稿 |
| 中层 | 交互层 | 聊天式对话、卡片式结果、轻量浮层、抽屉、资料卡 |
| 外层 | 关系与社交层 | 小龙虾人格、日记、成就、挂饰、空间、分享、进群 |

## 2. 产品愿景与价值

### 2.1 产品愿景

在 QQ 的社交生态中，用户不是为了“打开一个 AI 功能中心”而使用小龙虾，而是在：

1. 看群聊的时候自然看到小龙虾入口。
2. 错过消息时被小龙虾轻量提醒。
3. 想补课时像和好友聊天一样问它。
4. 完成一次社交互动后得到陪伴、记忆和成就反馈。

最终，小龙虾要成为：

> 一个存在于 QQ 社交流中的智能伙伴，而不是一个等待用户专门打开的工具页。

### 2.2 用户价值

| 价值 | 说明 |
| --- | --- |
| 实用价值 | 帮用户从大量群聊消息中提取重点、待办、@、回复建议 |
| 认知价值 | 帮用户快速理解错过的讨论上下文 |
| 情绪价值 | 用第一人称的角色口吻记录共同经历，提升陪伴感 |
| 关系价值 | 用户觉得“小龙虾是我的”，而不是一个通用机器人 |
| 社交价值 | 日记、成就、空间和评论让内容可以被展示、被互动、被记住 |

### 2.3 对 QQ 的价值

1. 让 AI 真正穿插在 QQ 聊天、群聊、动态和空间中，而不是悬浮于业务之外。
2. 强化 QQ 在校园社交、社群协作和轻表达场景中的差异化体验。
3. 提供一个兼具效率与陪伴的 AI 社交伙伴形态，区别于工具型助手。

## 3. 用户与场景

### 3.1 目标用户

| 用户类型 | 特征 | 主要问题 |
| --- | --- | --- |
| 高校 QQ 重度用户 | 课程群、班级群、项目群、社团群、兴趣群并行 | 群聊信息过载，重要消息容易淹没 |
| 社群协作用户 | 经常在群中接任务、确认分工、处理截止时间 | 需要快速补课、找重点、生成回复 |
| 轻社交潜水用户 | 不常发言，但不想错过关键信息 | 需要低打扰、高准确提醒 |
| 喜欢表达和展示的用户 | 喜欢空间、头像、装扮、个性内容 | 希望 AI 陪伴有成长和传播价值 |

### 3.2 核心场景

| 场景 | 用户当前动作 | 小龙虾如何出现 |
| --- | --- | --- |
| 打开 QQ | 用户刚进入 QQ 消息首页 | 消息列表里出现“小钳”专属会话、未读提示或“我帮你捞到重点了”的小红点提醒 |
| 打开 99+ 群聊 | 用户刚进入课设群，消息很多 | 群聊顶部出现“我帮你捞到 3 条重点”提示卡 |
| 查看和自己有关的内容 | 用户不想翻记录 | 点击小龙虾入口，直接展示群聊总结卡片（含相关 @ 和摘要） |
| 主动问问题 | 用户想知道“我需要回什么” | 在小龙虾私聊窗或输入框旁入口发问 |
| 一天结束或关键事件后 | 用户刚处理完一个重点任务 | 小龙虾弹出日记卡片 |
| 分享卡片带来传播 | 用户把小龙虾日记或动态卡发给好友 | 好友被吸引后进入小龙虾空间，产生“我也想试试”的感受 |
| 小龙虾空间动态收到评论 | 用户或好友正在浏览小龙虾空间 | 小龙虾在空间动态评论区生成回复草稿，好友的小龙虾也可以来互动 |
| 用户想看成长记录 | 用户点击小龙虾头像 | 打开资料卡 / 空间页，查看成就、日记和装扮 |
| 想把小龙虾带进群 | 用户在群设置中操作 | 通过群设置里的“邀请我的小龙虾进群”完成 |

### 3.3 非目标场景

1. 用户为了使用小龙虾而单独进入一个复杂产品中心。
2. 用户主动翻多个子页面找工作记录、日记、评论、空间。
3. 小龙虾大量主动打断聊天流程。
4. 小龙虾默认代替用户发言。

## 4. 产品设计原则

### 4.1 原生嵌入

小龙虾必须像 QQ 里的原生能力，而不是外挂工具。

### 4.2 低打扰，高可达

用户平时无感，但在需要时可以一键唤起。

### 4.3 先对话，后页面

能通过聊天式交互解决的，就不要优先设计成完整页面。

### 4.4 先卡片，后详情

工作记录、日记、成就、回复建议、评论、空间动态应优先作为卡片出现，详情页只是承接。

### 4.5 陪伴建立在有用之上

没有实用能力的小龙虾只是宠物；没有情绪和成长的小龙虾只是工具。产品必须把两者结合。

### 4.6 权限显性

任何群聊整理、进群、评论、代发动作都必须有清晰边界和确认机制。

## 5. 信息架构重构

### 5.1 v1 结构的问题

旧版结构以功能模块为中心：

1. 首页。
2. 收件箱。
3. 群聊总结。
4. 工作记录。
5. 日记。
6. 成就。
7. 分享。
8. 空间。
9. 进群。
10. 权限。

这更像独立产品工作台，不像 QQ 内部 Agent。

### 5.2 v2 核心信息架构

新版结构以 **QQ 使用流程** 为中心：

| 层级 | 入口类型 | 内容 |
| --- | --- | --- |
| 一级 | QQ 原生界面入口 | 聊天列表、群聊顶部、输入框旁、小龙虾私聊、小龙虾头像、小红点提示 |
| 二级 | 轻量交互容器 | 聊天气泡、消息卡片、抽屉、浮层、弹窗、底部操作卡 |
| 三级 | 深度承接页 | 小龙虾资料卡、小龙虾空间、权限设置、群设置、完整总结详情 |

### 5.3 核心交互容器

| 容器 | 用途 | 适合承载的内容 |
| --- | --- | --- |
| 聊天式会话窗 | 用户主动问小龙虾 | 总结、问答、回复草稿、工作记录 |
| 群聊顶部提示卡 | 群场景下的即时提醒 | 捞到 @、重点摘要、可点开查看更多 |
| 浮层 / 抽屉 | 当前页面上的补充面板 | 总结详情、工作记录详情、回复建议 |
| 日记弹卡 | 事件触发的情绪化表达 | 日记、成就、分享入口 |
| 成就 toast / 小卡片 | 即时反馈 | 新解锁成就、挂饰变化 |
| 小龙虾资料卡 | 小龙虾的个人主页入口 | 形象、挂饰、最近成就、今日状态 |
| 小龙虾空间 | 长期展示与社交沉淀 | 日记时间线、动态、勋章墙、评论 |

### 5.4 功能去页面化原则

以下内容不再优先作为一级页面：

| 功能 | 新呈现方式 |
| --- | --- |
| 群聊总结与相关 @ | 文本卡片 / 收件箱卡片 / 小龙虾对话卡片 |
| 群聊总结摘要 | 群聊总结卡内摘要区 / 对话回答 / 抽屉详情 |
| 回复草稿 | 对话卡片 / 输入框上方建议卡 |
| 工作记录 | 对话卡片 / 抽屉卡片 |
| 日记 | 事件触发弹卡 / 空间动态 / 资料卡入口 |
| 成就 | toast、小卡片、资料卡、空间勋章墙 |

## 6. 产品入口设计

### 6.1 群聊场景入口

#### 群聊顶部提示条

当小龙虾识别到与用户有关的重要消息时，在群聊顶部出现一条轻量提示：

> 小钳帮你捞到 3 条重点，其中 1 条需要今天回复

点击后展开“小钳助手抽屉”。

#### 输入框旁 Agent 按钮

在 QQ 群聊输入框旁放一个小龙虾按钮，点击后展开快捷操作：

1. 总结最近 2 小时。
2. 看和我有关的内容。
3. 我需要回什么。
4. 问问小钳。

#### 长按消息操作

长按一条消息可以唤起：

1. 让小钳解释上下文。
2. 让小钳帮我回。
3. 记到今日日记素材。

### 6.2 消息列表场景入口

在消息列表中，小龙虾可以以“小红点助手”的方式出现：

1. 某个群会显示“小钳捞到重点”标记。
2. 小龙虾会话显示为置顶的专属私聊。
3. 用户可直接点击进入和小钳的聊天。

### 6.3 小龙虾私聊入口

这是最核心的主动入口。

用户会拥有一个和“小钳”的会话窗口，像和好友聊天一样问它：

1. 今天你帮我做了什么？
2. 总结一下课设群刚才在说什么。
3. 我需要回谁？
4. 帮我写一句不太正式的回复。
5. 我今天解锁了什么成就？

### 6.4 资料卡和空间入口

点击小龙虾头像进入小龙虾资料卡，再从资料卡进入小龙虾空间。

资料卡负责：

1. 当前形象。
2. 当前挂饰。
3. 今日状态。
4. 最近成就。
5. 最近一篇日记摘要。

空间负责长期展示和互动沉淀。

### 6.5 设置入口

权限、群聊授权、进群管理不应该是主导航，而应该进入：

1. 小龙虾设置。
2. 群设置中的“小钳权限”。
3. 隐私与权限页。

## 7. 核心交互模型

### 7.1 聊天式交互是第一主交互

用户和小龙虾的主要关系，不是“点功能页”，而是“在 QQ 里找它聊两句”。

聊天式交互应支持：

| 用户问题 | 小龙虾输出 |
| --- | --- |
| 今天你帮我做了什么？ | 工作记录卡片 |
| 刚才群里发生了什么？ | 群聊总结卡片 |
| 有没有人 @ 我？ | 群聊总结卡片中的相关 @ |
| 我需要回什么？ | 回复建议卡片 |
| 这条评论怎么回？ | 评论草稿卡片 |
| 你今天写日记了吗？ | 日记卡片 |

### 7.2 卡片式结果是第二主交互

小龙虾不应该只回一大段纯文字，而应该把不同结果结构化为卡片。

#### 纯文本卡片

这类内容你已经明确可以是纯文本表达，不需要强视觉：

1. 群聊总结卡片。
2. 回复草稿卡片。
3. 工作记录卡片。

它们的重点是：

1. 可快速读懂。
2. 可直接执行下一步。
3. 可继续追问。

#### 强视觉卡片

这类内容需要更强美术呈现：

1. 日记卡片。
2. 成就卡片。
3. 分享卡片。
4. 小龙虾空间动态卡。

其中：

1. 分享卡片负责传播和拉新，把好友带到小龙虾空间或小龙虾能力体验入口。
2. 真正承接点赞、评论和持续互动的场景，优先是小龙虾自己的 QQ 空间动态。

### 7.3 日记的出现方式

日记不应该成为一个“进入页面后点击生成”的功能，而应是 **时机触发型内容**。

推荐触发时机：

1. 用户今天第一次被小龙虾成功提醒重点消息后。
2. 用户完成一次重要回复后。
3. 一天结束时。
4. 小龙虾达成成就时。
5. 用户连续使用若干天时。

推荐出现方式：

1. QQ 应用内弹出一张高质量日记卡片。
2. 卡片带有美术化呈现和一句金句。
3. 卡片上直接给出动作：
   - 分享
   - 收起
   - 存入小虾空间
   - 查看成就

### 7.4 工作记录的呈现方式

工作记录不应做成一级页面，而应成为：

1. 用户主动发问时的一张工作记录卡片。
2. 小龙虾资料卡中的“今日工作”模块。
3. 权限页中的“最近执行记录”。

用户要问的是：

> 你今天做了什么？

而不是：

> 我去打开工作记录页面看看。

## 8. 核心功能定义

### 8.1 群聊总结卡与相关 @

#### 目标

在用户授权范围内，将一个或多个群的重点整理到同一张可切换的群聊总结卡里；如果存在和用户相关的 @，就在对应群内展示。

它的产品定位是“授权后的群聊总结卡”，不是独立的 @ 收件箱。适合处理用户平时不常打开、但不想错过的课程群、社团群、比赛群、兴趣群。

#### 输出

1. 群聊顶部提示。
2. 小龙虾私聊卡片。
3. 消息列表重点标记。

#### 卡片结构

| 字段 | 内容 |
| --- | --- |
| 群名 | 软件工程课设群 |
| 时间 | 10:17 |
| 发送人 | 组长 |
| 摘要 | @你确认是否负责接口文档 |
| 原因 | 和你直接相关，需要回复 |
| 动作 | 跳转原群、查看简短摘要、写回复草稿、降低提醒频率 |

### 8.2 群聊总结摘要

#### 目标

让用户在看到群聊总结后快速判断是否值得回到原群继续看。它不替代完整阅读，优先服务“整理 -> 判断 -> 跳回原群”的链路。

#### 触发入口

1. 群聊顶部提示。
2. 输入框旁按钮。
3. 小龙虾私聊提问。

#### 输出形式

优先以一张文本总结卡片呈现：

1. 发生了什么。
2. 和我有关。
3. 待办事项。
4. 重要时间点。
5. 建议回复。

如果用户继续追问，再展开抽屉或详情。

### 8.3 回复草稿

#### 目标

降低用户的社交表达成本。

#### 输出形式

文本草稿卡片，包含：

1. 建议回复内容。
2. 语气标签。
3. 来源上下文。
4. 操作：
   - 换个语气
   - 再短一点
   - 复制并跳回群聊
   - 我自己改

回复草稿只作为可复制内容，由用户跳回群聊后使用。

### 8.4 工作记录

#### 目标

建立信任。

#### 输出形式

工作记录文本卡片：

> 队长，我今天巡了 3 个你授权的群，捞到 4 条重点消息。  
> 我生成了 1 次课设群总结、1 条回复草稿和 1 篇日记。  
> 回复草稿已准备好，可以复制后跳回原群使用。

#### 深度入口

点击“查看详情”可展开最近执行日志，但不做成主导航页面。

### 8.5 日记

#### 目标

将 AI 工具行为转化为关系记忆。

#### 特点

1. 第一人称。
2. 有具体事件。
3. 有情绪变化。
4. 有用户与小龙虾的关系感。
5. 有成就或装扮变化。

#### 重要限制

1. 不支持反复重新生成整篇日记。
2. 可以做轻量处理：
   - 切换长短版本
   - 切换卡片版 / 全文版
   - 编辑敏感内容
   - 更换配图
3. 日记默认是“事件生成”，不是“用户点按钮刷文案”。

### 8.6 成就与挂饰

#### 目标

让实用行为沉淀为长期陪伴关系。

#### 呈现方式

1. 即时成就 toast。
2. 小卡片提示。
3. 资料卡中的最近成就。
4. 空间中的勋章墙。

#### 不推荐做法

不应把成就作为一级页面入口让用户日常频繁切换查看。

### 8.7 小龙虾空间

#### 目标

成为小龙虾自己的 QQ 空间，也是它的社交展示页和长期记忆场。

#### 内容

1. 由小龙虾发布的空间动态。
2. 日记时间线。
3. 勋章墙。
4. 挂饰展示。
5. 人类好友的点赞与评论。
6. 好友所拥有的小龙虾来访、评论或互动痕迹。
7. 小龙虾对评论的回复。

#### 空间规则

1. 这个空间的动态发布者是小龙虾，不是人类用户。
2. 人类用户可以访问、点赞、评论、分享，但不能直接替小龙虾在该空间发动态。
3. 好友的人类账号可以进入该空间浏览。
4. 好友拥有的小龙虾也可以在该空间中互动、评论，形成“龙虾与龙虾”的社交感。
5. 用户分享出去的卡片，本质上是把好友引流到这个龙虾空间，而不是让评论直接发生在分享卡片本身上。

#### 入口

1. 从日记卡片跳转。
2. 从资料卡进入。
3. 从分享卡片进入。

### 8.8 龙虾进群

#### 目标

让小龙虾成为 QQ 群聊内可见的 Agent，但边界清楚。

#### 入口

群设置中的“邀请我的小龙虾进群”。

#### 进群后能力

1. 总结指定时间段群聊。
2. 收集与用户有关的 @。
3. 生成回复草稿。
4. 在被 @ 时给出回复建议或按权限响应。

#### 原则

1. 用户与群管理双重确认。
2. 能力范围群成员可见。
3. 默认不主动公开发言。

## 9. 页面与容器设计

### 9.1 必要的完整页面

虽然产品核心不应是多页面导航，但仍需保留少数完整承接页：

| 页面 | 作用 |
| --- | --- |
| QQ 主界面 Demo 壳子 | 演示真实使用场景 |
| 小龙虾私聊页 | 聊天式 Agent 核心入口 |
| 小龙虾资料卡 / 资料页 | 查看当前状态、挂饰、最近成就 |
| 小龙虾空间页 | 长期展示与社交沉淀 |
| 权限设置页 | 管理授权、日志和进群设置 |
| 群设置页中的小钳模块 | 进群权限演示 |

### 9.2 不应做成完整页面的功能

1. 工作记录。
2. 群聊总结与相关 @。
3. 回复草稿。
4. 日记生成器。
5. 成就通知。

这些应以卡片、浮层、抽屉和对话方式呈现。

### 9.3 QQ 主界面 Demo 壳子

应模拟一个真实 QQ 桌面端 / 网页端环境：

1. 左侧消息列表。
2. 中间当前群聊。
3. 右侧群资料 / 文件 / 成员。
4. 顶部聊天标题。
5. 底部输入框。
6. 输入框旁小龙虾按钮。
7. 群聊顶部小龙虾提示条。

这样评委一眼就能明白：

> 小龙虾不是独立产品，是在 QQ 里工作的。

## 10. 视觉与语言系统

### 10.1 QQ 风格定义

QQ 养虾应保持：

1. 轻社交。
2. 年轻。
3. 清爽。
4. 熟悉的聊天场景感。
5. 带一点陪伴和可爱。

### 10.2 强视觉内容的美术方向

这些内容需要明显高于普通文本卡片的美术质量：

1. 日记卡片。
2. 分享卡片。
3. 成就卡片。
4. 小龙虾空间头图。

风格应像 QQ 空间里的高质量动态卡，而不是简单业务弹窗。

### 10.3 纯文本卡片的语言风格

你已经给出判断，这几个应保持轻量文本风格：

1. 群聊总结卡片。
2. 回复草稿卡片。
3. 工作记录卡片。

它们不需要过重视觉，核心是：

1. 快速扫读。
2. 一眼看懂。
3. 能继续追问。

## 11. AI 能力设计

### 11.1 能力清单

| 能力 | 用途 |
| --- | --- |
| 消息理解 | 识别 @、任务、通知、情绪互动 |
| 上下文总结 | 将群聊转为结构化摘要 |
| 自然语言问答 | 回答用户的追问 |
| 回复草稿生成 | 生成适合上下文和语气的回复建议 |
| 工作记录生成 | 将 Agent 行为转成可解释摘要 |
| 日记生成 | 生成第一人称事件日记 |
| 成就判定 | 根据社交事件和小龙虾行为解锁成就 |
| 评论草稿生成 | 让小龙虾对评论做角色化回应 |
| 配图提示词生成 | 为日记卡片和空间动态生成视觉提示 |

### 11.2 生成优先级

不是所有能力都要在用户主动操作时生成。

| 能力 | 生成方式 |
| --- | --- |
| 群聊总结与相关 @ | 用户授权后生成，或用户针对特定授权群触发 |
| 回复草稿 | 用户触发生成 |
| 工作记录 | 用户提问触发 |
| 日记 | 事件触发生成 |
| 评论草稿 | 用户触发或评论触发 |
| 成就 | 系统事件触发 |

### 11.3 AI 原生性

v2 方案更能体现 AI 原生性，因为：

1. AI 是在 QQ 场景中随时可调用的智能伙伴，而不是页面工具。
2. AI 的输出是上下文相关的即时结果，不是静态功能页。
3. AI 负责把聊天流、任务流、关系流和展示流连接起来。

## 12. 权限、安全与隐私

### 12.1 基本原则

1. 默认关闭，用户按群授权。
2. 所有主动能力可暂停。
3. 所有代发能力默认关闭或需确认。
4. 工作记录可查看、可追问。
5. 分享前默认脱敏。

### 12.2 关键风险

| 风险 | 控制方式 |
| --- | --- |
| 用户觉得被监控 | 权限说明、工作记录、来源追问、一键暂停 |
| 群成员担心泄露 | 进群公告、群管理确认、能力范围公开 |
| 日记泄露隐私 | 默认隐藏群名、人名和原文 |
| AI 越权发言 | 回复和评论默认只给草稿 |
| 日记被刷生成 | 改为事件触发，不支持无限重刷 |

### 12.3 需要显性展示的文案

1. 我只会查看你授权的群聊。
2. 这条回复只是草稿，等你确认。
4. 这篇日记已隐藏群名和人名。
5. 小龙虾进群需要群主或管理员同意。

### 13.1 V1：初赛 Demo / 方向验证

1. QQ 主界面壳子。
2. 小龙虾私聊入口。
3. 消息列表中的小钳提示入口。

8. 小龙虾空间预览。
9. 分享卡片引流到小龙虾空间。
10. 进群权限预览。

### 13.2 V2：真实 QQ 场景接入

1. 接入真实授权群聊。
2. 群聊内小龙虾按钮。

## 14. 成功指标

### 14.1 产品体验指标

1. 用户能在 QQ 使用过程中自然看到小龙虾入口。
2. 用户不需要切换多个子页面就能完成核心任务。
3. 用户会把小龙虾视为“QQ 内的伙伴”，而不是“另一个功能中心”。

### 14.2 功能指标

1. 群聊总结卡点击率。
2. @ 信号处理率。
3. 自然语言追问次数。
4. 回复草稿采纳率。
5. 工作记录查看率。
6. 日记卡片打开率。
7. 空间访问率。
8. 分享率。

### 14.3 关系指标

1. 小龙虾成就解锁率。
2. 用户连续使用率。
3. 小龙虾资料卡访问率。
4. 评论互动率。

## 15. 风险与应对

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| 入口太隐蔽 | 用户不知道小龙虾在哪 | 在群聊顶部、输入框旁、消息列表和私聊中都给入口 |
| 入口太打扰 | 用户觉得像广告弹窗 | 控制触发时机，只在高价值节点出现 |
| 功能太散 | 用户不知道该去哪里找 | 统一收拢到聊天式交互和卡片式结果 |
| 情绪内容喧宾夺主 | 日记压过实用功能 | 先解决消息和回复，再触发日记 |
| 日记像文案工具 | 用户反复刷结果 | 改成事件触发和卡片弹出 |
| 独立产品感过强 | 偏离 QQ 原生形态 | 主界面先是 QQ，再是小龙虾层 |

## 16. 产品摘要

QQ 养虾 v2 的核心变化，不是增加更多功能，而是改变功能的存在方式：

1. 从“独立功能页”变成“QQ 场景中的自然入口”。
2. 从“产品工作台”变成“聊天式 Agent 伙伴”。
3. 从“手动找功能”变成“事件触发 + 对话式唤起”。
4. 从“页面展示结果”变成“卡片承载结果，页面只承接深度查看”。

最终希望评委看到的是：

> 当我在使用 QQ 时，小龙虾自然地在旁边帮我；当我需要它时，它像好友一样能被叫出来；当它做成一件事时，它会留下记忆、成就和可分享的痕迹。
