# QQ 养虾 v2 分阶段实施计划

更新时间：2026-04-30

## 总体原则

本项目采用分阶段实施。每阶段都应产出可运行、可体验、可验收的小版本。用户验收后，再进入下一阶段。

不一次性实现完整 Demo。

讨论中形成的新产品信息、协作约定和阶段验收结论，应及时同步到 `implementation_state.md`、`stage_plan.md` 或 `handoff_notes.md`。

视觉资产采用协作方式推进：用户可以使用 `gpt-image-2` 生成图片；实现侧需要提前明确小龙虾形象、动画关键帧、日记卡、成就挂饰和空间头图的规格需求。

阶段 1 已完成。下一步不直接扩功能页面，优先讨论 `OpenClaw` 最小框架、数据库和真实 AI adapter，让后续能力从真实数据流推进，同时保留 mock 降级。

阶段 2 原本规划为“小龙虾认养流程”，但认养已经并入阶段 1 完成。后续不再单独执行阶段 2。

阶段 1.5 已完成。它不是一次性完成后就不再维护的模块，而是后续阶段持续演进的数据与 AI 底座。阶段 3 之后每个功能阶段都必须同时考虑：

1. 前端交互。
2. OpenClaw API。
3. SQLite 数据记录。
4. 真实 AI adapter。
5. mock fallback。
6. 演示降级策略。
7. 能力注册。
8. 工具调用。
9. 审查策略。
10. 事件、工作记录和记忆写入。

## 阶段 1：QQ 主界面 + 小龙虾首次出现

状态：已完成，等待用户体验反馈。

### 目标

用户打开 Demo 后，第一眼看到的是 QQ 主界面，并在普通使用过程中看到小龙虾动画入口。

### 工作内容

1. 实现 QQ 主界面：
   - 左侧会话列表
   - 中间聊天窗口
   - 右侧群资料
   - 底部输入框
2. 默认展示一个普通群聊场景。
3. 在合适时机出现小龙虾动画入口。
4. 小龙虾入口文案示例：
   - `我还没有开始帮你看消息。要不要先认识一下，把我养在 QQ 里？`
5. 点击入口进入认养流程。

### 第一阶段范围边界

第一阶段先只追求“打开 QQ 后自然遇到小龙虾”的感觉稳定，不实现完整功能闭环。

应包含：

1. QQ 主界面底座。
2. 普通聊天/群聊环境。
3. 小龙虾 Q 版可爱形象入口。
4. 首次出现动画或关键帧模拟。
5. 点击入口后进入认养流程。
6. 认养完成后进入小龙虾私聊界面，先露出正确的信息架构。

暂不包含：

1. 群聊总结。
2. @ 消息收集。
3. 工作记录。
4. 日记惊喜。
5. 小龙虾空间。
6. 成就与挂饰完整系统。

1. Q 版可爱小龙虾主形象，QQ 宠物式轻拟人化，可以有简单衣服或小配饰，透明背景。
2. 首次出现动画关键帧 3-5 张，能表达“小龙虾从 QQ 界面里探出来”。
3. 认养流程里的小龙虾欢迎图或轻表情图。

4. 小龙虾入口是自然触发，不像强广告。已实现延迟出现的右下角入口，待用户体验判断。

6. 小龙虾视觉风格偏 Q 版可爱、胖嘟嘟、QQ 宠物式轻拟人化，用户愿意接触。当前为 CSS 绘制占位形象，待用户决定是否替换为图片资产。

### 阶段 1 边界修正

1. 认养前的小龙虾不能声称已经捞到 @、总结群聊或处理过消息。

5. 形象允许拟人化，但应控制为 QQ 宠物式轻拟人化：小龙虾仍是主体，可以穿简单小马甲、背带、围巾、胸牌或挂饰；不要做人类比例、真人脸、完整人形四肢或成人化助手。

## 阶段 1.5：OpenClaw 最小框架 + 真实 AI Adapter

状态：已完成。后续作为持续演进底座维护。

### 目标

把当前静态交互 Demo 升级为“有最小真实数据流 + 真实 AI 能力入口”的产品骨架。

### 核心原则

1. 不推翻当前 `apps/qq-lobster-v2` 前端。
2. 新增 `apps/openclaw-api` 作为最小后端。
3. 数据库先用本地 SQLite。
4. 前端通过 adapter 访问 OpenClaw，不直接访问数据库或真实 AI。
5. AI 使用 OpenAI 兼容接口，支持通过 `baseURL + apiKey + model` 适配 DeepSeek、Kimi 2.6 等主流模型。
6. 所有真实 AI 调用必须有 mock fallback，确保演示稳定。
7. 暂不接真实 QQ 登录、真实 QQ 消息和真实发送能力。

### Agent 底座结构

OpenClaw 不只是数据库和 AI adapter。它应维持一个可控的 Agent 流程：

```text
用户输入 / 系统事件
  -> capability resolver
  -> permission checker
  -> tool planner
  -> guardrails pre-check
  -> tool executor
  -> AI generator
  -> guardrails post-check
  -> preview / confirmation
  -> event log / work log / agent output / memory
```

核心模块：

1. `Memory`：小龙虾身份、权限、行为、能力和长期偏好。
2. `Capabilities`：小龙虾能做什么、触发方式、权限要求、风险等级和输出类型。
3. `Tools`：小龙虾靠什么执行能力，包括 mock 工具和后续真实工具。
4. `Review`：前置和后置审查，保证权限、安全、隐私和发布确认。
5. `AI Adapter`：统一接 OpenAI 兼容模型，失败时 fallback。
6. `Streaming`：小龙虾聊天框应支持流式输出，但必须和审查、卡片状态、确认动作协调。

### 第一版真实 AI 范围

1. 小龙虾私聊对话。
2. 群聊总结 / @ 信息理解。
3. 工作记录生成。
4. 小龙虾第一人称日记生成。

暂不优先做：

1. 图片生成。
2. 小龙虾空间复杂评论 AI。
3. 长期记忆系统。
4. 多用户实时共养。

### 最小数据对象

1. `users`：Demo 用户。
2. `lobsters`：小龙虾资料，包含名字、称呼、性格、兴趣。
3. `groups`：模拟 QQ 群。
4. `messages`：模拟群聊消息。
5. `permissions`：群聊授权、主动感知/总结、回复草稿权限；`collect_mentions` 字段保留为底层兼容，不作为独立外露权限。
6. `checkins`：打卡进度。
7. `work_logs`：小龙虾做过什么，支持追问。
8. `agent_outputs`：总结、回复草稿、日记、空间动态等 AI 输出。
9. `ai_requests`：真实 AI 请求日志，不保存密钥。
10. `capabilities`：能力注册表，记录能力 key、说明、触发示例、权限、风险等级、输出类型和是否启用。
11. `tools`：工具注册表，记录工具 key、输入输出 schema、所需权限、风险等级、是否需要确认和是否有 mock。
12. `tool_runs`：工具调用记录，支撑工作记录、日记素材和调试。
13. `review_policies`：审查策略，记录前置/后置审查规则。
14. `review_results`：审查结果记录，记录通过、拦截、重写或预览确认。
15. `memories`：显式长期记忆，第一版以结构化资料和规则写入为主。

### 记忆分层

1. 身份记忆：用户是谁、小龙虾是谁、称呼、性格、兴趣、等级、挂饰。
2. 权限记忆：哪些群授权、哪些能力可用、哪些动作必须确认。
3. 行为记忆：小龙虾做过什么、生成过什么、失败或 fallback 过什么。
4. 能力记忆：小龙虾能做什么、不能做什么、做之前需要什么条件。
5. 语义记忆：用户偏好、长期兴趣、互动习惯，第一版不做复杂 RAG。

### 第一版工具

1. `read_mock_group_messages`
2. `collect_mentions`
3. `summarize_messages`
4. `generate_reply_draft`
5. `generate_work_log`
6. `generate_diary`
7. `generate_space_post_preview`

### 第一版审查策略

1. `check_group_permission`：禁止读取未授权群。
2. `block_auto_send`：禁止自动替用户发送群消息。
3. `require_preview_for_high_risk`：高风险动作必须预览确认。
4. `desensitize_share_output`：分享卡片和空间动态发布前脱敏。
5. `block_unverified_real_qq_claim`：禁止声称已经执行真实 QQ 操作。

### 最小 API 草案

1. `GET /health`
2. `GET /api/bootstrap`
3. `POST /api/adoption`
4. `POST /api/permissions`
5. `POST /api/checkins/:key/complete`
6. `GET /api/work-logs`
7. `POST /api/ai/chat`
8. `POST /api/ai/summarize-group`
9. `POST /api/ai/generate-work-log`
10. `POST /api/ai/generate-diary`

### AI 配置草案

```text
OPENCLAW_AI_PROVIDER=openai-compatible
OPENCLAW_AI_BASE_URL=
OPENCLAW_AI_API_KEY=
OPENCLAW_AI_MODEL=
OPENCLAW_AI_TIMEOUT_MS=30000
OPENCLAW_AI_FALLBACK=mock
```

### 验收标准草案

1. 不配置 AI key 时，前端仍可 mock 运行。
2. 配置 OpenAI 兼容接口后，小龙虾私聊可以返回真实 AI 文本。
3. 群聊总结可以读取模拟群聊消息并生成结构化摘要。
4. 工作记录和日记能基于用户操作记录生成。
5. 所有 AI 输出都能记录到数据库，标记来源为 `real-ai` 或 `mock-fallback`。

### 持续演进规则

阶段 1.5 只建立最小底座。后续阶段推进时，不能只做前端页面；每个新能力都需要同步补齐对应的 API、数据表/记录、能力注册、工具调用、审查结果、AI adapter、mock fallback 和演示降级。

流式输出规则：

1. 普通闲聊、说明、低风险文本可以流式展示。
2. 群聊总结、工作记录等结构化卡片可以先流式展示生成过程，最终卡片需在完整输出和审查后落库。
3. 回复草稿、空间动态、分享卡片、日记等涉及确认、脱敏或发布链路的内容，操作按钮必须等后置审查通过后才可用。
4. 流式失败时，展示短错误态并切换到 mock fallback，不阻断演示。

## 阶段 2：已并入阶段 1，不再单独执行

### 目标

阶段 2 原目标是“小龙虾认养流程”。该能力已经在阶段 1 中完成，因为第一次体验必须从 QQ 主界面自然出现并完成认养，不能把认养拆到后续阶段。

### 已完成内容

1. 认养流程包括：
   - 小龙虾命名，默认“小钳”
   - 小龙虾对用户称呼，默认“队长”
   - 性格选择
   - 兴趣爱好初始化
2. 性格选项：
   - 毒舌但靠谱
   - 温柔陪伴
   - 社恐观察
   - 高冷秘书
   - 热血队友
3. 兴趣选项：
   - AI 工具
   - 课程项目
   - 游戏社群
   - 校园活动
   - 表情包
4. 认养完成后进入小龙虾私聊 / 主交互界面。

### 结论

1. 阶段 2 不再作为待执行阶段。
2. 后续认养相关工作只作为 OpenClaw 数据持久化、真实 AI 个性化上下文和小龙虾资料维护的一部分继续演进。
3. 若进入阶段 1.5，认养资料需要写入数据库，并成为真实 AI prompt 的基础上下文。

## 阶段 3：小龙虾主交互界面

### 目标

实现用户上传草图中的三栏结构。

### 结构

```text
左侧：聊天窗口索引
中间：小龙虾聊天主窗口
右侧：小龙虾直观可视化界面 + 引导任务
```

### 工作内容

1. 左侧展示会话索引。
2. 中间承载所有聊天输入输出。
3. 右侧展示：
   - 小龙虾形象
   - 当前性格
   - 当前挂饰
   - 当前新手任务
   - 已完成任务
   - 去空间
   - 去权限
4. 所有核心功能先从聊天窗口触发。
5. 小龙虾聊天输出需要支持流式输出，模拟真实 AI 正在思考和打字。
6. 流式消息需要有状态：生成中、已完成、审查中、可操作、失败 fallback。

### 验收标准

1. 用户知道接下来应该和小龙虾聊天。
2. 右侧是引导和状态，不抢主交互。
3. 后续所有卡片都能放入中间聊天流。
4. 用户能感受到小龙虾在实时回复，而不是一次性吐出静态文本。

### OpenClaw 对接要求

1. 读取数据库中的小龙虾资料、称呼、性格、兴趣。
2. 小龙虾私聊消息通过统一 service/store 写入，后续可同步记录到 `agent_outputs` 或聊天记录表。
3. 小龙虾私聊对话优先走真实 AI adapter；失败时返回 mock fallback。
4. API 需支持流式返回或模拟流式分片；前端只消费统一 stream adapter。
5. 流式输出完成后再写入最终 `agent_outputs`，并记录是否通过后置审查。

### 执行记录

状态：已完成，等待用户体验反馈。

1. 认养后的小龙虾主交互界面已收敛为三栏结构：
   - 左侧：聊天窗口索引。
   - 中间：小龙虾聊天主窗口，承载用户输入和小龙虾输出。
   - 右侧：小龙虾形象、性格、兴趣、状态、打卡、奖励、去空间和去权限入口。
2. 小龙虾私聊消息已通过 Zustand store 统一写入，后续卡片可以继续放进同一聊天流。
3. 已新增统一 stream adapter：前端通过 OpenClaw `/api/ai/chat` 获取最终输出，再模拟流式分片展示。
4. 流式消息已展示 `生成中`、`审查中`、`已完成`、`fallback`、`失败` 等状态。
5. 第一次聊天完成后会同步完成 `first_lobster_chat` 打卡，并调用 OpenClaw checkin API；后端不可用时仍保留本地状态。
6. OpenClaw 后端已有 `/api/ai/chat` 的 `agent_outputs`、`ai_requests` 和 `work_logs` 记录链路，本阶段未额外扩展数据库表。
7. 已通过验证：
```text
apps/qq-lobster-v2: npm run lint
apps/qq-lobster-v2: npm run build
apps/openclaw-api: npm run smoke
```

## 阶段 4：新手任务引导系统

### 目标

通过“打卡”一步步引导用户学习小龙虾能力，并用奖励解锁承接成长感。

### 打卡顺序

1. 第一次和小龙虾对话。
2. 处理一次群聊提醒。
3. 第一次查看工作记录。
4. 第一次让小龙虾发龙虾空间动态。
5. 第一次在龙虾空间评论回复。

“处理一次群聊提醒”是一个完整功能闭环，不拆成右侧连续小打卡；内部仍包含设置群聊权限、生成群聊总结卡、查看卡片、跳回原群、展开摘要和写回复草稿，并继续写入 OpenClaw 日志。

### 验收标准

1. 每完成一步，自动解锁下一步。
2. 用户不会一次看到所有功能。
3. 打卡可以服务最终录屏。
4. 打卡越多，能解锁挂饰、徽章、空间装扮等奖励预览。

### OpenClaw 对接要求

1. 打卡状态写入 `checkins`。
2. 奖励解锁记录写入奖励或成就相关数据表。
3. 每次完成打卡时写入一条 `work_logs`，便于后续回答“小龙虾今天做了什么”。
4. 后端不可用时，前端使用本地 store mock 兜底。

## 阶段 5：权限与信息获取

状态：已完成，等待用户体验反馈。

### 目标

让用户自然理解如何授权一个或多个具体群，并让小龙虾主动感知这些群里的重点、@ 信号和可跳回来源。

本阶段的产品表达重点是“主动巡群提醒”，不是“再做一个手动群总结”。QQClaw 适合用户授权后巡看那些平时不常打开、但又不想错过的课程群、社团群、比赛群和兴趣群。

### 工作内容

1. 用户在聊天中问：
   - `帮我看看这个群有没有值得注意的`
   - `有没有人 @ 我`
2. 小龙虾提示需要授权具体群，可多选。
3. 弹出权限设置卡。
4. 用户选择群聊和能力：
   - 主动感知并总结群聊
   - 生成回复草稿
5. 授权完成后，小龙虾生成一张群聊感知卡，卡片内包含简短摘要、@ 信号和来源跳转。

### 验收标准

1. 权限是由需求自然引导出来。
2. 不是用户自己去找设置。
3. @ 消息不作为独立卡片或全局感知能力出现，只作为所选群聊感知结果里的高优先级信号。
4. 群聊感知卡支持一键跳转来源。

### OpenClaw 对接要求

1. 群聊授权、主动感知/总结、回复草稿权限写入 `permissions`；底层 `collect_mentions` 只作为 `summarize_group` 内部信号提取工具兼容保留。
2. 模拟群聊和消息从 OpenClaw bootstrap 接口读取；后端不可用时使用前端 mock 数据。
3. 权限变更写入 `work_logs`，用于透明展示小龙虾做过什么。
4. 真实 AI 调用必须检查授权范围，不能总结未授权群。
5. 如果后续接腾讯内部能力，QQClaw 可调用元宝 AI 的群消息理解/摘要能力；但 QQClaw 负责授权、主动巡群、提醒判断、宠物化表达和跳回原群。

### 执行记录

1. 小龙虾私聊右侧“设置群聊权限”不再只是完成打卡按钮，会在中间聊天流自然弹出权限设置卡。
2. 权限卡支持多选具体授权群和两类外露能力：主动感知并总结群聊、生成回复草稿；日记素材不在权限卡中提前暴露。
3. 授权保存优先写入 OpenClaw `/api/permissions`；多选时仍按群逐条保存权限记录，后端不可用时保留前端本地权限状态作为 fallback。
4. 授权后会为每个已选群通过 `/api/ai/summarize-group` 生成群聊感知卡；卡片内把 @ 我的消息作为提醒信号展示，不再生成独立 @ 信息卡。
5. 来源跳转会回到 QQ 群聊视图并高亮原消息，同时 QQ 主界面保留回到小龙虾的入口，避免流程断掉。
6. 群聊感知绑定用户在权限卡里选择的具体群集合，会逐群检查 `summarize_group` 权限，再返回摘要、@ 信号、来源消息 ID；失败时使用本地 mock fallback。
7. OpenClaw 侧权限更新写入 `permissions`、`work_logs`、`events` 和 permission memory；工具调用继续写入 `tool_runs`、`review_results`、`work_logs` 和 behavior memory。
8. 已验证未授权群感知被 `check_group_permission` 阻断，授权后摘要和 @ 信号合并返回，并记录 AI 输出来源。

## 阶段 6：核心文本卡片

### 目标

实现核心实用能力。

核心文本卡片要服务“发现 -> 提醒 -> 跳回原群”的主动感知链路，不把群聊总结做成替代用户阅读的长报告。

### 卡片类型

1. 群聊感知卡：包含简短摘要、@ 信号、为什么值得看和来源跳转。
2. 群聊摘要展开态：作为群聊感知卡的属性或后续状态，不再独立于 @ 消息。
3. 回复草稿卡。
4. 工作记录卡。

### 要求

1. 文本清楚。
2. 来源明确。
3. 有一键跳转。
4. 能继续追问。
5. 回复草稿必须标注：发送前需要确认。
6. 巡群提醒卡应短而克制，只说明为什么值得看，并提供跳转原群、查看简短摘要、降低提醒频率等操作。

### 验收标准

1. 用户能看到小龙虾真的处理了 QQ 信息流。
2. 不需要重视觉，效率优先。

### OpenClaw 对接要求

1. @ 信号从 `messages` 和当前用户 ID 过滤得到，并挂在群聊感知卡上。
2. 群聊感知/摘要、回复草稿通过真实 AI adapter 生成；后续可接元宝 AI 的群消息理解/摘要能力。
3. AI 输出写入 `agent_outputs`，并标记来源 `real-ai` 或 `mock-fallback`。
4. 工作记录写入 `work_logs`，支持后续追问。
5. 失败时返回 mock 卡片，不阻断演示。

### 执行记录

1. 已实现核心文本卡片：群聊感知短提醒、同卡摘要展开态、回复草稿卡和工作记录卡。
2. 群聊感知卡默认只说明为什么值得看，保留跳回原群、查看简短摘要、写回复草稿、降低提醒频率和继续追问入口。
3. 回复草稿接入 OpenClaw `/api/ai/reply-draft`，通过 `generate_reply_draft` 工具和预览确认审查，卡片明确标注发送前需要确认。
4. 工作记录接入 OpenClaw `/api/ai/generate-work-log`，返回 AI 文本和最新 `work_logs`，前端以工作记录卡展示。
5. 已验证阶段 6 smoke：回复草稿保留来源消息 `m-002`，`previewRequired: true`，工作记录返回最新日志；失败时仍可使用本地 fallback。

## 阶段 7：隐藏日记惊喜

### 目标

通过隐藏触发制造惊喜和情绪连接。

### 触发条件

用户完成以下若干任务后：

1. 已认养。
2. 已授权群聊。
3. 已查看 @ 信息。
4. 已总结一次群聊。
5. 已生成一次回复草稿。

### 触发表现

小龙虾动画弹出：

```text
队长，我刚刚偷偷写了一篇日记。
要看看吗？
```

用户点击后回到小龙虾聊天界面。

### 展示方式

1. 聊天框里小龙虾发日记。
2. 同时出现一张美术化日记卡片。
3. 卡片包含小龙虾形象、金句、今日成就。
4. 看完后解锁“日记入口”。
5. 之后用户可以在小龙虾交互界面中查看历史日记。

### 验收标准

1. 第一次日记是惊喜，不是按钮功能。
2. 不支持反复重新生成。
3. 看完后沉淀到小龙虾空间。

### OpenClaw 对接要求

1. 日记触发条件基于 `checkins`、`work_logs` 和 `agent_outputs` 判断。
2. 日记文本通过真实 AI adapter 生成，使用小龙虾第一人称。
3. 生成结果写入 `agent_outputs`，后续可沉淀为空间动态。
4. 已触发日记状态写入数据库，避免刷新后重复弹出第一次惊喜。
5. AI 失败时使用预设日记 mock fallback。

### 执行记录

1. 已实现隐藏日记触发：后端基于已认养、已授权群聊、已出现 @ 信号、已生成群聊总结和已生成回复草稿判断是否可触发。
2. 日记生成接入 OpenClaw `/api/ai/generate-diary`，会读取 `checkins`、`work_logs` 和 `agent_outputs` 作为素材，通过 `generate_diary` 工具和 AI adapter 生成第一人称日记。
3. 第一次触发结果写入 `agent_outputs`、`work_logs` 和 diary memory；`GET /api/diary/hidden-first` 可查询状态，`POST /api/diary/hidden-first/reveal` 会记录已查看，避免刷新后重复弹出第一次惊喜。
4. 前端已在完成摘要和回复草稿链路后自动触发隐藏惊喜弹层，用户点击“要看看”后进入小龙虾聊天流，展示日记正文和美术化日记卡片。
5. 日记卡包含小龙虾形象、金句和今日成就；看完后右侧解锁“日记入口”，可查看历史日记。
6. 已验证阶段 7 smoke：触发前 `canTrigger: true`，生成后 `triggered: true`，重复调用不重新生成，revealed 后 `unlocked: true`；真实 AI 不可用时使用 mock fallback。

## 阶段 8：小龙虾空间

### 目标

承接长期展示和社交传播。

### 空间规则

1. 动态由小龙虾发布。
2. 人类可以访问、点赞、评论、分享。
3. 人类不能替小龙虾在空间发动态。
4. 好友的小龙虾可以互动或评论。

### 初赛需要做

1. 一条小龙虾日记动态。
2. 一条成就动态。
3. 好友评论。
4. 好友龙虾评论。
5. 小龙虾回复草稿。

### 验收标准

1. 分享卡片负责引流到空间。
2. 评论发生在空间动态下。
3. 空间像小龙虾自己的 QQ 空间。

### OpenClaw 对接要求

1. 空间动态、点赞、评论需要独立数据记录，后续可新增 `space_posts`、`space_comments`。
2. 小龙虾发布动态时，应关联来源 `agent_outputs` 或 `work_logs`。
3. 好友评论和好友龙虾评论可先 seed 为 mock 数据。
4. 小龙虾回复草稿通过真实 AI adapter 生成，发送前保持预览确认。

### 完成记录

1. OpenClaw 已新增 `space_posts`、`space_comments` 和 `space_interactions`，空间动态、评论、点赞和分享均可独立落库。
2. 小龙虾空间动态接入 `/api/ai/generate-space-post`，通过 `generate_space_post_preview` 工具、AI adapter、`agent_outputs`、`work_logs`、审查结果和 space memory 留痕。
3. 空间会展示一条小龙虾日记动态和一条成就动态；好友评论和好友小龙虾评论由 seed 数据生成。
4. 人类只能在空间里点赞、评论、分享，不能替小龙虾发动态；动态卡明确标注“小龙虾发布”。
5. 小龙虾评论回复接入 `/api/ai/space-comment-reply`，保持 `previewRequired: true`，并记录为小龙虾作者的空间评论。
6. 前端新增“龙虾空间”视图和右侧空间入口；日记查看后可“存入空间/进入空间”，空间内可点赞、评论、分享和触发小龙虾回复评论。
7. 已验证阶段 8 smoke：生成空间动态返回 `previewRequired: true`，空间包含日记动态和成就动态，点赞/评论/分享均落库，小龙虾评论回复保持预览态并写入动态评论区。
